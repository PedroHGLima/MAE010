\section*{Introdução}
Este relatório apresenta o desenvolvimento e a análise de seis exercícios práticos utilizando a biblioteca MPI para programação paralela. O objetivo do trabalho é aplicar conceitos de comunicação ponto-a-ponto, algoritmos de redução, comunicações coletivas e tipos de dados derivados, avaliando o comportamento dos processos no sistema operacional e o desempenho computacional em termos de escalabilidade e convergência.

Neste trabalho, os códigos serão executados no ambiente WSL2 em um computador equipado com um processador \href{https://www.intel.com.br/content/www/br/pt/products/sku/236789/intel-core-i7-processor-14700kf-33m-cache-up-to-5-60-ghz/specifications.html}{Intel Core i7 14700KF}, com um total de 28 threads.

\section{Exercício 1: Variantes do Hello World}
Ambos os códigos deste exercício foram desenvolvidos com foco em observar os identificadores de processos (PIDs) e como o MPI se comporta.

\subsection{Introspecção em nível de SO}
O modelo de programação do MPI funciona disparando processos independentes no sistema operacional. Nesta seção, verificou-se como esses processos são alocados no hardware ao executar o programa com 8 processos através do comando \verb|mpiexec -n 8 ./hello_os|. Diferente de um modelo de threads que compartilham memória, o MPI cria processos isolados, o que é confirmado na saída do terminal, pois cada rank possui um Identificador de Processo (\textit{PID}) próprio.

\begin{lstlisting}[language=bash, caption=Exemplo de uma execução do programa, basicstyle=\ttfamily\footnotesize]
    $ mpiexec -n 8 ./hello_os
    Hello from rank 1/8 -- PID = 6571, CPU = 8
    Hello from rank 2/8 -- PID = 6572, CPU = 1
    Hello from rank 3/8 -- PID = 6573, CPU = 14
    Hello from rank 4/8 -- PID = 6574, CPU = 12
    Hello from rank 5/8 -- PID = 6575, CPU = 4
    Hello from rank 7/8 -- PID = 6577, CPU = 2
    Hello from rank 6/8 -- PID = 6576, CPU = 0
    Hello from rank 0/8 -- PID = 6570, CPU = 4
\end{lstlisting}

Ao realizar execuções consecutivas, notou-se que os PIDs são gerados de forma sequencial (de 6570 a 6577). Isso acontece porque o MPI cria os processos um logo após o outro rapidamente. Também observou-se que a coluna da CPU muda a cada execução e não segue a ordem dos ranks. Como a alocação é feita com base no instante em que o processo é lançado e o sistema operacional gerencia muitas tarefas ao mesmo tempo, podemos considerar a distribuição dos processos nos núcleos do processador sendo aleatória.

\subsection{Variante em anel}
Nesta variante, os processos foram organizados para se comunicar em uma estrutura de anel, onde cada rank envia sua saudação para o vizinho da direita e recebe do vizinho da esquerda. No cálculo para descobrir o vizinho da esquerda do rank 0, é feito \verb|(my_rank - 1 + comm_sz) % comm_sz|. A soma do número total de processos antes de aplicar o operador de resto (\verb|%|) serve para garantir que o resultado dentro do parêntese seja positivo, já que em C o operador de resto mantém o sinal do dividendo e retornaria um índice negativo inválido (\verb|-1|).

\begin{lstlisting}[language=Bash, caption=Exemplo de execução do programa em anel, basicstyle=\ttfamily\footnotesize]
    $ mpiexec -n 4 ./hello_ring
    Rank 0 received: Greetings from rank 3!
    Greetings from rank 0!
    Greetings from rank 1!
    Greetings from rank 2!
\end{lstlisting}

Uma implementação simples onde todos os processos tentam receber uma mensagem antes de enviar causaria um \textit{deadlock}, pois todos ficariam esperando uma ação que nunca aconteceria. Para evitar esse problema, optou-se pelo uso da função \verb|MPI_Sendrecv|. Essa primitiva do MPI resolve o impasse realizando o envio e a recepção de dados ao mesmo tempo de forma coordenada. Assim, todos os processos recebem a mensagem do anel com sucesso. No fim do código, o rank 0 recebe via \verb|MPI_Recv| as mensagens que os demais processos encaminharam para centralizar a exibição dos resultados na tela.

\section{Exercício 2: Regra do Trapézio Generalizada}

\subsection{Balanceamento de carga para n arbitrário}
Neste exercício, a regra do trapézio para o cálculo numérico de uma integral foi alterada para permitir a execução quando o número de trapézios ($n$) não é um múltiplo exato do número de processos ($p$). O algoritmo calcula o resto da divisão de $n$ por $p$ e distribui um trapézio adicional para os primeiros processos, ajustando o deslocamento (\textit{offset}) de cada um para evitar a omissão ou a duplicidade de trechos do intervalo de integração.

A validação foi feita com a função $f(x) = \sin(x)$ no intervalo de $0$ a $\pi$, utilizando 3 processos e 10 trapézios. Nessa configuração, o resto da divisão é igual a 1, fazendo com que o processo 0 calcule 4 trapézios, enquanto os processos 1 e 2 calculem 3 trapézios cada. A execução com a entrada informada teve como resultado $\int_0^\pi f(x)dx \approx 1.98$ em um tempo de $9.50\,\mu s$.

\begin{lstlisting}[language=bash, caption={Validação com p = 3 e n = 10}, basicstyle=\ttfamily\footnotesize]
    $ echo "0 3.1415 10" | mpiexec -n 3 ./mpi_trap_generalized
    With n = 10, integral of f on [0, 3.1415] = 1.98352
    T_p (p=3): 9.50300e-06 seconds
\end{lstlisting}

\subsection{Estudo de escalabilidade forte}
O estudo de escalabilidade forte foi realizado fixando o número total de trapézios em $n = 10^7$ e variando a quantidade de processos em $p \in \{1, 2, 4, 8\}$. O tempo sequencial de referência ($T_s$) foi obtido executando o próprio código paralelo com $p = 1$, incluindo os custos de inicialização e finalização do ambiente MPI. A medição foi feita com a função \verb|MPI_Wtime| ao redor do bloco de computação e da redução, coletando o maior tempo entre os processos por meio de uma chamada \verb|MPI_Reduce| com o operador \verb|MPI_MAX|.

Os tempos de execução ($T_p$), os valores de speedup ($S_p = T_s / T_p$) e a eficiência ($E_p = S_p / p$) obtidos estão listados na Tabela 1.

\begin{table}[H]
\centering
\caption{Métricas de escalabilidade forte para $n = 10^7$}
\begin{tabular}{cccc}
\hline
Processos ($p$) & Tempo $T_p$ ($ms$) & Speedup $S_p$ & Eficiência $E_p$ \\ \hline
1 ($T_s$)       & 39.56              & -             & -                \\
2               & 23.87              & 1.66          & 0.83             \\
4               & 13.43              & 2.95          & 0.74             \\
8               & 7.06               & 5.60          & 0.70             \\ \hline
\end{tabular}
\end{table}

A partir dos dados coletados, observa-se que a eficiência cai abaixo de 0.8 a partir de $p = 4$ processos. Esse comportamento ocorre porque, com o aumento do número de processos trabalhando sobre o mesmo volume total de dados, o tempo gasto com a comunicação e com a sincronização passa a representar uma parcela maior do tempo total de execução. Além disso, o ganho com a divisão do trabalho computacional começa a sofrer com a disputa por recursos compartilhados, como a banda de memória do processador.

\subsection{Estudo de convergência}
Neste item, o número de processos foi fixado em $p = 4$ para avaliar o comportamento da aproximação da integral com o aumento do número de trapézios, utilizando $n \in \{64, 256, 1024, 4096, 16384\}$ e limite superior em $\pi = 3.1415$.

\begin{lstlisting}[language=bash, caption={Resultados de convergência com p = 4}, basicstyle=\ttfamily\footnotesize]

    T_p (p=4): 1.79960e-05 seconds

    T_p (p=4): 1.66120e-05 seconds
    With n = 1024, on [0, 3.1415] = 1.999998427068573

Com o aumento de $n$, o tamanho do espaçamento $h$ diminui, reduzindo o erro do método numérico. Os resultados mostram que a aproximação converge para o valor esperado de 2.00000 em 5 casas decimais a partir de $n = 1024$.

Observa-se também que o tempo de execução medido não é estritamente crescente em relação a $n$. Como esses tamanhos de problema são muito pequenos, o custo puramente computacional é insignificante. A variação de tempo observada reflete predominantemente o atraso das mensagens de comunicação do MPI (\textit{overhead}) e as flutuações do escalonador do sistema operacional, que ofuscam o tempo de cálculo da integral.

\section{Exercício 3: Soma Paralela}
O processo 0 gerou um vetor de 12000 números aleatórios e distribuiu fatias iguais para cada rank via \verb|MPI_Send|. Cada processo recebeu os dados com \verb|MPI_Recv|, calculou uma soma local e retornou o valor parcial para o processo 0 realizar o acúmulo final. A validação contra o cálculo serial apresentou um erro relativo de $3.33 \times 10^{-15}$, causado pela reordenação das adições na aritmética de ponto flutuante.

\begin{lstlisting}[language=bash, caption={Saída da soma paralela}, basicstyle=\ttfamily\footnotesize]
    Parallel Sum = 6.005344909739839e+03
    Serial Sum   = 6.005344909739819e+03
    Relative Error = 3.331845837289520e-15
\end{lstlisting}

A implementação ponto-a-ponto utilizou cerca de 15 linhas de código para o controle manual dos laços de envio e recebimento de mensagens. Com o uso de comunicações coletivas, esse bloco é substituído por apenas duas chamadas: \verb|MPI_Scatter| e \verb|MPI_Reduce|.

\section{Exercício 4: Prática de Comunicação Coletiva}

No primeiro item, usamos a função \verb|MPI_Gather| para agrupar as saudações de todos os processos em um \textit{buffer} alocado no processo 0, substituindo o laço de recepções manuais. A operação coletiva organiza as de forma sequencial de acordo com o rank de origem.

\begin{lstlisting}[language=bash, caption={Saída do hello\_gather}, basicstyle=\ttfamily\footnotesize]
    Greetings from process 0 of 4!
    Greetings from process 1 of 4!
    Greetings from process 2 of 4!
    Greetings from process 3 of 4!
\end{lstlisting}

No segundo item, a distribuição de um vetor com 12000 números do tipo \verb|double| foi feita através do comando \verb|MPI_Scatter|. Cada processo realizou a busca do menor e do maior elemento local e, por fim, enviou esses dados para o processo 0 por meio de duas chamadas da função \verb|MPI_Reduce|, parametrizadas com as operações \verb|MPI_MIN| e \verb|MPI_MAX|. Os valores obtidos foram comparados com uma busca sequencial e a validação confirmou os limites globais.

\begin{lstlisting}[language=bash, caption={Saída do minmax}, basicstyle=\ttfamily\footnotesize]
    Global Minimum (MPI): 8.12784e-05 | Serial: 8.12784e-05
    Global Maximum (MPI): 9.99998e-01 | Serial: 9.99998e-01
    Check: Success
\end{lstlisting}

\section{Exercício 5: Soma Paralela de Vetores}
Os vetores $x$ e $y$ foram distribuídos entre os processos utilizando a função \verb|MPI_Scatter|, permitindo que cada rank realizasse a soma local do seu bloco correspondente ($z = x + y$). Na primeira versão, o vetor resultante $z$ foi coletado apenas no processo 0 por meio da função \verb|MPI_Gather|, que validou o resultado contra o cálculo sequencial. Na segunda versão, a operação foi substituída por \verb|MPI_Allgather|, fazendo com que todos os processos terminassem a execução com o vetor $z$ completo em memória, o que é verificado pelo valor impresso da última posição do vetor ($full\_z[N-1] = 35997.0$) em todos os ranks.

\begin{lstlisting}[language=bash, caption={Saída da soma paralela de vetores}, basicstyle=\ttfamily\footnotesize]
    $ mpiexec -n 4 ./mpi_vecadd_gather
    Check gather: Success

    $ mpiexec -n 4 ./mpi_vecadd_allgather
    Rank 0 over. full_z[N-1] = 35997.0
    Rank 1 over. full_z[N-1] = 35997.0
    Rank 2 over. full_z[N-1] = 35997.0
    Rank 3 over. full_z[N-1] = 35997.0
\end{lstlisting}

A versão com \verb|MPI_Allgather| é útil em algoritmos iterativos onde todos os processos necessitam do resultado completo da etapa atual para realizar os cálculos da próxima iteração, como em métodos numéricos para sistemas lineares ou simulações de N-corpos. O uso do \verb|MPI_Allgather| nessas situações é preferível pois elimina a necessidade de realizar uma chamada subsequente de \verb|MPI_Bcast| pelo processo raiz, reduzindo as etapas de sincronização e a latência da comunicação.

\section{Exercício 6: Um Tipo de Dado Derivado MPI}
Para transmitir uma estrutura (\verb|struct Student|), é criado um tipo derivado MPI utilizando a primitiva \verb|MPI_Type_create_struct| e a macro \verb|offsetof|. Na primeira versão, o processo 0 realizou a difusão dos dados com uma única chamada \verb|MPI_Bcast|. Na segunda versão, o envio foi dividido em três chamadas independentes de \verb|MPI_Bcast|, uma para cada campo da estrutura. Ambas as abordagens entregaram um conteúdo final idêntico em todos os processos.

\begin{lstlisting}[language=bash, caption={Saída do tipo derivado e das três chamadas}, basicstyle=\ttfamily\footnotesize]
    $ mpiexec -n 4 ./student_struct
    Rank 0 received: Alice, Grade: 9.5, ID: 12345
    Rank 1 received: Alice, Grade: 9.5, ID: 12345
    Rank 2 received: Alice, Grade: 9.5, ID: 12345
    Rank 3 received: Alice, Grade: 9.5, ID: 12345

    $ mpiexec -n 4 ./student_three_bcasts
    Rank 1 received: Alice, Grade: 9.5, ID: 12345
    Rank 2 received: Alice, Grade: 9.5, ID: 12345
    Rank 3 received: Alice, Grade: 9.5, ID: 12345
    Rank 0 received: Alice, Grade: 9.5, ID: 12345
\end{lstlisting}

A comparação mostra que a versão com tipo derivado reduz o número de chamadas da biblioteca de três para uma. Em termos de desempenho, essa abordagem reduz o tempo de CPU gasto com o custo de invocação das funções do MPI e permite que os dados sejam lidos e copiados em um bloco contíguo de memória, diminuindo o overhead de processamento quando comparado à execução de transmissões isoladas para cada variável.
